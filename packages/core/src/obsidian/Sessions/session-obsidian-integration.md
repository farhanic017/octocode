# Obsidian Vault Integration for Cross-Session Context

**Date:** 2026-06-22
**Project:** 1octo-code-main (octocode)
**Goal:** Make agents able to use Obsidian vaults fully so they maintain context across sessions

## Summary

Implemented full Obsidian vault integration into octocode's agent system. Agents now automatically read project notes and session history from a configured Obsidian vault at session start, and write session summaries back when done. This gives agents persistent memory of prior work, decisions, and ongoing tasks across sessions.

## What Was Built

### 1. Config Module (`packages/core/src/config/obsidian.ts`)
New config schema for vault settings:
- `vault_path` - absolute path to Obsidian vault
- `enabled` - toggle integration
- `sync_sessions` - auto-sync session summaries to vault
- `note_folder` - subfolder for OctoCode notes (default: OctoCode)
- `max_context_notes` - max notes to inject as context (default: 20)

### 2. Vault Service (`packages/core/src/obsidian/vault.ts`)
Core vault reader/writer:
- Reads all .md notes from vault's OctoCode/ folder
- Parses YAML frontmatter and extracts titles
- Matches notes to current project by name, path, or frontmatter
- Searches notes by query terms
- Writes notes back to vault
- Builds context string from project notes + recent session notes
- Uses Effect-based FSUtil for all I/O (no raw fs)
- Gracefully handles vault unavailable (returns empty, no crash)

### 3. Session Sync (`packages/core/src/obsidian/session-sync.ts`)
Writes session data back to vault:
- Generates Markdown session notes with YAML frontmatter (project, tokens, files, agent, model)
- Generates per-project index notes linking all sessions
- Uses Obsidian [[wikilinks]] for cross-references
- Auto-detects vault path and note folder from config

### 4. SystemContext Source (`packages/core/src/system-context/obsidian-context.ts`)
Injects vault context into agents:
- Registers `core/obsidian-vault` context key
- Loads project notes + recent session notes
- Injects as system context so agents see prior work automatically
- Gracefully handles errors (empty context, no crash)

### 5. Wiring
- Added `obsidian` field to `Config.Info` schema
- Wired `ObsidianVault.layer` into `LocationServiceMap` dependencies
- Wired `ObsidianSystemContext.layer` into location layer composition
- Both services available through the standard Effect layer chain

## Architecture Decisions
- Used direct filesystem access for vault reads (agents work locally, no MCP needed)
- All vault reads return `never` error type (catch all errors gracefully)
- Vault integration is opt-in: only activates when vault_path is configured or default vault exists
- SystemContext source catches all defects to prevent vault issues from crashing agents
- Session sync writes to vault's OctoCode/Sessions/ and OctoCode/Projects/ folders

## How It Works (Flow)
1. Agent starts session → SystemContext registers vault notes as ambient context
2. Agent reads vault → Sees project notes, prior session summaries, decisions
3. Session ends/compacts → ObsidianSessionSync writes summary back to vault
4. Next session → Agent automatically sees the updated vault context
